# Waqfa Reporting Standard

## Purpose

This standard defines where generated project reports are stored, how they are named, and when temporary storage is allowed.

## Canonical Location

All persistent reports generated by development, bootstrap, validation, audit, infrastructure, Docker, or PocketBase tooling must be stored under:

```text
00_Documentation/09_Reports/
```

Required categories:

```text
00_Documentation/09_Reports/
├── archive/
├── bootstrap/
├── development/
├── docker/
├── infrastructure/
├── pocketbase/
└── runtime/
```

Git does not track empty directories, so each category may contain a `.gitkeep` file until the first real report is committed.

## Naming Convention

Use:

```text
YYYY-MM-DD-<report-name>.md
```

Examples:

```text
2026-07-24-development-runtime-audit.md
2026-07-24-bootstrap-report.md
2026-07-24-docker-validation.md
2026-07-24-pocketbase-schema-audit.md
```

When more than one report of the same type is created on the same date, append a descriptive suffix rather than overwriting prior evidence.

## Required Metadata

Each report should begin with:

```md
# <Report Title>

**Date:** YYYY-MM-DD
**Branch:** <branch-name>
**Commit:** <commit-sha>
**Generated By:** <script-or-tool>
**Status:** PASS | WARN | FAIL
```

## Script Requirements

Any script that produces information with future diagnostic, audit, validation, or operational value must:

1. Write the report to the matching directory under `00_Documentation/09_Reports/`.
2. Create the target directory with `mkdir -p` before writing.
3. Use Markdown for human-readable reports unless a machine-readable format is explicitly required.
4. Preserve prior reports instead of silently overwriting them.
5. Print the final report path after successful generation.
6. Exit with a non-zero status when validation fails.

Example:

```bash
REPORT_DIR="00_Documentation/09_Reports/runtime"
REPORT="$REPORT_DIR/$(date +%F)-development-runtime-audit.md"

mkdir -p "$REPORT_DIR"
```

## Use of `/tmp`

`/tmp` is permitted only for transient intermediate files that have no value after the command or script completes.

Do not use `/tmp` as the final destination for:

- Environment audits
- Bootstrap reports
- Docker validation reports
- PocketBase schema or migration reports
- Security or configuration audits
- CI reproduction reports
- Development runtime diagnostics intended for later review

## Version-Control Guidance

Commit reports when they establish project state, provide audit evidence, explain an incident, or support a pull request.

High-volume or repetitive local reports may remain uncommitted when they contain no durable value. Sensitive data and secrets must never be committed.

## Definition of Done

A reporting-related change is complete when:

- [ ] The report is stored under the canonical location.
- [ ] The filename follows the naming convention.
- [ ] Required metadata is included.
- [ ] Secrets and sensitive values are redacted.
- [ ] The generating script prints the saved path.
- [ ] Related documentation is updated in the same pull request.
