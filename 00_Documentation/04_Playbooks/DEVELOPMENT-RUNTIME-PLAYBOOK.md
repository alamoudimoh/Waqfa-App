# Development Runtime Playbook

## Purpose

A reusable implementation guide for establishing a consistent local development runtime for modern TypeScript monorepos.

## Outcomes

A completed runtime bootstrap should provide:

- Reproducible Node.js and package-manager versions.
- A documented workspace structure.
- Environment-variable templates without secrets.
- Deterministic installation, bootstrap, and verification commands.
- Development containers with explicit networking and health behavior.
- Persistent validation reports.
- A clear Definition of Done.

## Recommended Sequence

### 1. Establish Runtime Contracts

Define:

- Node.js version.
- Package-manager version.
- Package-manager workspace file.
- Root TypeScript configuration.
- Canonical repository checkout path when absolute mounts are required.
- Application ports.
- Local timezone.

Example files:

```text
package.json
pnpm-workspace.yaml
tsconfig.json
.nvmrc
.editorconfig
.gitignore
```

### 2. Define Environment Templates

Create committed templates rather than committed secrets:

```text
.env.example
.env.development.example
.env.production.example
```

Rules:

- Never commit credentials, tokens, private keys, or production secrets.
- Document each variable's purpose.
- Use explicit service URLs and ports.
- Keep environment-specific values separated.

### 3. Add Bootstrap Automation

Create one command that prepares the development environment and another that verifies it.

Recommended layout:

```text
tools/bootstrap/bootstrap.sh
tools/bootstrap/verify.sh
```

Bootstrap responsibilities:

- Check required binaries.
- Check expected versions.
- Install dependencies.
- Create required local directories.
- Validate required external resources.
- Generate a persistent Markdown report.

Verification responsibilities:

- Typecheck packages and applications.
- Build required applications.
- Validate configuration files.
- Check expected directories and environment templates.
- Exit non-zero on failure.

### 4. Define Development Containers

Use a dedicated development Compose file:

```text
infrastructure/compose/compose.development.yaml
```

Recommended standards:

- Explicit `container_name` values.
- Explicit application ports.
- Health checks for every long-running service.
- `restart: unless-stopped` where appropriate.
- JSON log rotation.
- Explicit timezone.
- Absolute bind mounts only when the checkout path is an intentional infrastructure contract.
- External networks for shared development infrastructure.
- Reuse existing databases and backends instead of creating duplicate services.

### 5. Validate External Dependencies

Before startup, verify:

- Required Docker networks exist.
- Shared backend containers are running.
- Expected DNS/container names resolve on the network.
- Required host ports are available.
- The repository exists at any configured absolute mount path.

### 6. Start the Runtime

Typical sequence:

```bash
corepack enable
pnpm install
pnpm bootstrap
pnpm verify

docker compose \
  -f infrastructure/compose/compose.development.yaml \
  config

docker compose \
  -f infrastructure/compose/compose.development.yaml \
  up -d
```

Then verify:

- Containers are running.
- Health checks pass.
- Each expected endpoint responds.
- Logs contain no startup loops or missing-environment errors.

## Reporting Standard

Store durable reports inside the repository documentation tree rather than `/tmp`.

Suggested structure:

```text
00_Documentation/09_Reports/
├── bootstrap/
├── development/
├── docker/
├── infrastructure/
└── runtime/
```

Naming convention:

```text
YYYY-MM-DD-<report-name>.md
```

Required metadata:

```md
# <Report Title>

**Date:** YYYY-MM-DD
**Branch:** <branch-name>
**Commit:** <commit-sha>
**Generated By:** <script-or-tool>
**Status:** PASS | WARN | FAIL
```

Use `/tmp` only for disposable intermediate files.

## Audit vs. Playbook

Use two distinct documents:

- **Audit:** Records the actual state, evidence, failures, and remaining work for one project and environment.
- **Playbook:** Describes the reusable method, sequence, standards, and checklist for future projects.

Do not treat a generic playbook as proof that a specific project runtime works.

## Common Failure Modes

- Starting a second database or backend when a shared development service already exists.
- Using undeclared ports that conflict with other projects.
- Depending on a Docker network that has not been created.
- Using relative bind mounts from an unpredictable working directory.
- Committing `.env` files containing secrets.
- Allowing verification scripts to report failure while exiting with status zero.
- Saving final audit evidence only under `/tmp`.
- Declaring runtime completion before builds, health checks, and endpoints are tested.

## Reusable Checklist

### Runtime

- [ ] Node.js version is pinned.
- [ ] Package-manager version is pinned.
- [ ] Workspace configuration is committed.
- [ ] TypeScript baseline is committed.
- [ ] Editor and ignore rules are committed.

### Environment

- [ ] Environment templates exist.
- [ ] No secrets are committed.
- [ ] Required ports are documented.
- [ ] Required external services are documented.

### Automation

- [ ] Bootstrap script exists.
- [ ] Verification script exists.
- [ ] Scripts exit non-zero on failure.
- [ ] Scripts print generated report paths.

### Containers

- [ ] Compose configuration validates.
- [ ] Container names are explicit.
- [ ] Health checks exist.
- [ ] Log rotation is configured.
- [ ] Timezone is explicit.
- [ ] External networks and services are verified.

### Validation

- [ ] Dependency installation succeeds.
- [ ] Typecheck succeeds.
- [ ] Build succeeds.
- [ ] Containers start.
- [ ] Health checks pass.
- [ ] Expected endpoints respond.
- [ ] Runtime audit records exact results.

## Definition of Done

The Development Runtime Bootstrap is complete only when:

- All declared runtime files are committed.
- Bootstrap and verification commands succeed on the target host.
- Compose configuration is valid.
- Required external dependencies are available.
- All runtime containers are healthy.
- Expected endpoints respond.
- Durable validation evidence is stored in the reporting directory.
- The project-specific audit status is `PASS`.

## Lessons Applied from Waqfa

- Keep shared development PocketBase separate and reuse it through an external network.
- Reserve application ports before application implementation.
- Treat the canonical checkout path as an explicit contract when using absolute mounts.
- Bootstrap frontend directories before expecting Compose application services to start.
- Separate repository-level completion from host-level runtime validation.
